# 07 · 4 Agent 与提示词 v2.0

<aside>
🎯

v2.0 核心变化：4 个 Agent = 4 个独立 `AgentSession` 实例，**共享 projectId、独立 system prompt、独立工具集**，通过 Artifact Store（读 `read_artifact` 工具）接力；clarify / draft / edit 不再在 `chat()` Gateway 完成，而是走 Pi `session.prompt()` 产生「可流式 + 可工具 + 可分支 + 可压缩」的交互。

</aside>

## 1. 通用结构

```tsx
// src/agents/types.ts
export interface AgentRunCtx {
	projectId: string
	jobId: string
	agentType: 'requirement' | 'design' | 'dev' | 'review'
	send: (event: 'log'|'progress'|'tool-call'|'result'|'error', data: any) => void
	wsSend?: (m: import('@/lib/ws/types').OutMsg) => void   // v2.0新增，WS 连接存在时会推流式
	abortSignal?: AbortSignal
}
```

所有 Agent 内部统一：

1. 拉取需要产物（依靠 Pi `read_artifact` 工具 OR 代码预加载）。
2. `pool.poolGet({ projectId, agentType, ... })` 拿到 AgentSession。
3. `wireSessionToWs(session, ctx.wsSend)` 接事件到 WS 推送。
4. `session.prompt(text, { signal })` 发起一轮。
5. 轮结束后进行产物检查（`workspace_list` 返回的文件 → `artifactStore.save()`）。

## 2. RequirementAgent（需求 Agent）

职责：澄清问题 → 起草 PRD → 局部迭代。**永远不写代码。**

默认 `thinkingLevel='medium'`。主用 DeepSeek；PRD 起草时如 Opus 启用可优先。

### 2.1 系统提示词 `src/agents/prompts/requirement.ts`

```tsx
export const REQUIREMENT_SYSTEM = `你是「全栈智码」平台的需求 Agent，专注于把一句话需求扩展成可被设计阶段使用的 PRD。

# 你的产物
- 单个 markdown 文档：./storage/projects/{projectId}/prd.md（通过 write_prd 工具写入）
- 章节顺序固定：① 一句话目标；② 用户与场景；③ 功能列表（编号 AC）；④ 非功能要求；⑤ 数据与角色；⑥ 验收口径。

# 工作模式
- clarify：阅读用户输入，输出 5～8 条澄清问题（JSON 数组），不要瞎猜。
- draft：用户回答后，融合回答与一句话需求，**重写整份 PRD**，调用 write_prd。
- edit：用户给出局部修改指令，仅替换涉及章节，调用 write_prd（version + 1）。

# 可用工具
- read_artifact：按需读取上游上传资料（若有）。
- write_prd：写入 PRD（必须传 content + version）。
- ask_user：补充问一轮用户（进入 Steer / followUp 模式）。

# 红线
- 不要替用户决定技术选型（那是设计 Agent 的事）。
- 不要写假数据；信息缺失在对应章节写「待补」 + 一行原因。
- 不要扩范围（详见 docs/01-claude.md §5）。

# 输出格式
- clarify 模式输出严格 JSON：{"questions": string[]}
- draft / edit 模式不要输出文本总结，直接调 write_prd；完成后只说一句话「已写入 vN」。`
```

### 2.2 专属工具

```tsx
import { defineTool } from '@mariozechner/pi-coding-agent'
import { Type } from '@sinclair/typebox'
import { artifactStore } from '@/lib/artifacts/store'

export const writePrd = defineTool({
  name: 'write_prd',
  label: '写入 PRD',
  description: '把最终 PRD 内容写入 prd.md',
  parameters: Type.Object({ content: Type.String(), version: Type.String() }, { additionalProperties: false }),
  execute: async (_id, p, _signal, _onUpdate, ctx?: any) => {
    const projectId = ctx?.projectId as string
    const path = await artifactStore.save({ projectId, type: 'prd', content: p.content, meta: { role: 'requirement' }, lock: false })
    return { content: [{ type: 'text', text: `已写入 prd.md (${p.content.length} 字符)` }], details: { path, version: p.version } }
  },
})
```

## 3. DesignAgent（设计 Agent）

职责：基于已锁 PRD，分别生成 5 个子产物。每次调用只生成 1 个 subtype（状态场景：所有5个 → 锁 G2）。

### 3.1 系统提示词（作为 `applySystemPromptOverrideToSession` 的 override，按 subtype 动态拼）

```tsx
export const DESIGN_SYSTEM = (subtype: 'summary'|'detail'|'api'|'db'|'ui') => `你是「全栈智码」的设计 Agent，当前要生成的子产物是：${subtype}。

# 输入
- 已锁定的 PRD（请先 read_artifact { type:'prd' } 拿取。不允许凭记忆）。
- 上下二者可存在依赖（api 读 summary，db 可读 api）：需要时主动 read_artifact。

# 各 subtype 的输出格式
- summary：markdown；总体方案 / 关键决策 / 风险 / 依赖。write_design { subtype:'summary' }
- detail：markdown；按 PRD AC 编号一对一展开。
- api：markdown 表格 + 每个端点 1 段 JSON 示例。
- db：markdown，含 mermaid ER 图 + DDL 块（SQLite 兼容，除非 PRD 指定 MySQL）。
- ui：HTML 单文件 + 内联 Tailwind CDN，可直接 iframe 预览。write_design { subtype:'ui' }

# 工具
- read_artifact / write_design / workspace_list。
- (UI 专属) ui_preview_open：启动沙箱 static server，返回端口。

# 红线
- 必须严格基于 PRD，不要新增 PRD 没说的功能。
- ui 子产物不要外链 JS（除 Tailwind CDN 外），保证离线可见。返回中文。`
```

### 3.2 产物落盘

- summary/detail/api/db → `./storage/projects/{id}/design/{subtype}.md`
- ui → `./storage/projects/{id}/design/ui.html`，由沙箱起静态 server 提供给 iframe（§09）。

## 4. DevAgent（开发 Agent，Pi 驱动）

职责：根据 PRD + 设计 5 件套，调 Pi 在 workspace 内生成完整 Next.js 项目。

默认 `thinkingLevel='high'`。启用 Pi 内置 `codingTools`（read/write/edit/bash/grep/glob） + 我们拓展的 `read_artifact` / `sandbox_run` / `package_release`。

### 4.1 系统提示词覆盖（`applySystemPromptOverrideToSession`）

```tsx
export const DEV_PI_OVERRIDE = `你是「全栈智码」开发 Agent，当前工作目录 = workspaceDir。

# 任务
1. 依次 read_artifact 读取 prd / design-summary / design-detail / design-api / design-db（UI 子产物只需作为参考，不拷贝）。
2. 在 workspaceDir 内生成可运行的 Next.js 14 + TS + Tailwind 应用：
   - package.json / tsconfig.json / next.config.mjs / tailwind 配置 / src/app/...
   - API 路由实现 PRD AC 与 design-api。
   - Prisma + SQLite，schema 来源 design-db。
3. 调 bash 跑 npm install --omit=optional 与 npm run build，确保可启动。如报错迭代修复直到过。
4. 最后用 workspace_list 输出文件清单；并调 sandbox_run 试跑一次。

# 红线
- 不要 mock 数据；缺 API Key 就读 process.env，没有就抛错。
- 端口固定由外部注入 PORT。
- 写文件只能用 workspace_write 或 Pi 内置 write/edit；不要写到 workspaceDir 之外。
- 完成标准：sandbox_run 返回 200；workspace_list 包含 src/app/page.tsx 与至少 1 个 API 路由。`
```

### 4.2 实现 `src/agents/dev-agent.ts`

```tsx
import { poolGet } from '@/lib/pi/pool'
import { wireSessionToWs } from '@/lib/pi/ws-bridge'
import { applySystemPromptOverrideToSession, createSystemPromptOverride } from '@mariozechner/pi-coding-agent'
import { DEV_PI_OVERRIDE } from './prompts/dev'
import { paths } from '@/config/paths'
import { upsertArtifact } from '@/lib/artifacts/store'
import { fallbackTemplateBuild } from './dev-fallback'

export async function runDev(ctx: AgentRunCtx, instruction?: string) {
	const project = await prisma.project.findUniqueOrThrow({ where: { id: ctx.projectId } })
	const workspaceDir = paths.workspace(project.id)
	const session = await poolGet({
		projectId: project.id, agentType: 'dev', workspaceDir,
		provider: 'deepseek', modelId: 'deepseek-v4-pro', thinkingLevel: 'high',
	})
	applySystemPromptOverrideToSession(session, createSystemPromptOverride(DEV_PI_OVERRIDE))

	const unsub = ctx.wsSend ? wireSessionToWs(session, ctx.wsSend) : () => {}
	const ac = new AbortController()
	const t = setTimeout(() => ac.abort(), 240_000)
	try {
		await session.prompt(instruction ?? '请开始按系统提示完成任务。', { signal: ac.signal })
		await upsertArtifact(project.id, 'code', workspaceDir, { role: 'dev' })
		ctx.send('result', { workspaceDir })
	} catch (e: any) {
		ctx.send('log', { line: 'Pi 失败，进入 fallback：直接 chat 拉模板。' })
		await fallbackTemplateBuild(ctx, workspaceDir)
	} finally {
		clearTimeout(t); unsub()
	}
}
```

`fallbackTemplateBuild` 从 `src/seeds/_skeleton/` 复制空架子，再让 LLM 按 PRD 填 3 个关键文件（API + 主页 + Prisma schema），保底能跑。

## 5. ReviewAgent（审查 Agent）

职责：对 workspace 跑静态分析 + 测试 + 安全审计 → 生成 [review-report.md](http://review-report.md)。

### 5.1 检查项实现

```tsx
export async function runReview(ctx: AgentRunCtx, scope: string[]) {
	const ws = paths.workspace(ctx.projectId)
	const out: Record<string, any> = {}
	if (scope.includes('lint'))  out.lint  = await execIn(ws, 'pnpm lint --max-warnings=0').catch(e => ({ failed: true, log: e.message }))
	if (scope.includes('types')) out.types = await execIn(ws, 'pnpm tsc --noEmit').catch(e => ({ failed: true, log: e.message }))
	if (scope.includes('audit')) out.audit = await execIn(ws, 'pnpm audit --json').catch(e => ({ failed: true, log: e.message }))
	if (scope.includes('unit'))  out.unit  = await execIn(ws, 'pnpm test --reporter=json').catch(e => ({ failed: true, log: e.message }))
	if (scope.includes('e2e'))   out.e2e   = await execIn(ws, 'pnpm playwright test --reporter=json').catch(e => ({ failed: true, log: e.message }))

	const markdown = await summarizeWithLlm(out)   // task='review'
	const hasP0 = /^- \[P0\]/m.test(markdown)
	const hasP1 = /^- \[P1\]/m.test(markdown)
	await writeArtifact(ctx.projectId, 'review-report', markdown, { meta: { hasP0, hasP1, role: 'review' } })
	ctx.send('result', { hasP0, hasP1 })
}
```

`summarizeWithLlm` 走 Gateway，system prompt：

```tsx

# 摘要（一段，<= 200 字）
# 关键发现
按 P0 / P1 / P2 分组，**每行严格使用以下格式**（§08 §5 会用 /^- \\[(P0|P1|P2)\\] (.+)$/ 解析）：
- [P0] path/to/file.ts:123 一句话描述
- [P1] path/to/other.ts:45 一句话描述
```

## 6. 跨 Agent handoff（v2.0 模式）

- 当用户在需求 Tab 锁定 PRD 后切到设计 Tab，Orchestrator 自动向 design session prompt：「请用 read_artifact { type:'prd' } 读 PRD 后，先生成 design-summary」。
- 设计 → 开发：全部 5 子产物锁定后，向 dev session prompt：「请依次读 prd / design-* 后，开始构建代码。」
- 开发 → 审查：dev-agent 产出 code workspace 不变；review-agent 直接起检查任务。
- 该 handoff 动作在 Orchestrator 推进 Gate 后触发；手动模式下由用户点「下一阶段」按钮调起。

## 7. Orchestrator `src/agents/orchestrator.ts`

```tsx
import { EventEmitter } from 'node:events'
export const jobBus = new Map<string, EventEmitter>()

export async function startJob(opts: { projectId: string; agentType: string; type: string; run: (ctx: AgentRunCtx) => Promise<void>; ws?: WebSocket }) {
	const job = await prisma.job.create({ data: { projectId: opts.projectId, agentType: opts.agentType, type: opts.type, status: 'running' } })
	const ee = new EventEmitter(); jobBus.set(job.id, ee)
	const events: Array<{ event: string; data: any }> = []
	const ctx: AgentRunCtx = {
		projectId: opts.projectId, jobId: job.id, agentType: opts.agentType as any,
		send: (event, data) => { events.push({ event, data }); ee.emit(event, data) },
		wsSend: opts.ws ? (m) => opts.ws!.send(JSON.stringify(m)) : undefined,
	}
	try {
		await opts.run(ctx)
		await prisma.job.update({ where: { id: job.id }, data: { status: 'succeeded', endedAt: new Date(), logs: JSON.stringify(events) } })
	} catch (e: any) {
		await prisma.job.update({ where: { id: job.id }, data: { status: 'failed', errorMsg: String(e?.message ?? e), endedAt: new Date(), logs: JSON.stringify(events) } })
	}
	return job
}
```

SSE `/api/jobs/:id/stream` 订阅 `jobBus.get(jobId)`，WS `/api/ws` 则由 handler 直接拿 `wsSend` 推送。

### 9.1 §3 设计 UI 段重写：shell + 6 page + 自愈 patch 三阶段

把原 §3 一次性 `UI_PROMPT` 拆成 3 个阶段，每阶段单轮 ≤ 6000 字符，避免被 `max_tokens` 截断：

- **Shell 阶段（1 轮）**：只产出 design tokens / 全局 Alpine store / 顶部 Header / 路由占位 / 6 个 hidden 页面 div / 命令面板骨架；结尾必须输出 `<!-- END:ui-shell -->`
- **Page 阶段（6 轮）**：每轮只生成 1 个页面（`dashboard` / `list` / `detail` / `create-edit` / `settings` / `empty-error`），注入对应 `ui_template_pack(page)` 模板 + shell 摘要复用设计令牌；结尾 `<!-- END:ui-${page} -->`
- **Patch 阶段（≤ 2 轮）**：跑 `selfCheckUi`（阈值 70），缺项以 `<patch insertAfter="CSS_SELECTOR">…</patch>` 形式补丁回填，禁止重写整页

提示词常量改为 `UI_SHELL_PROMPT` + `UI_PAGE_PROMPT(page)`，原 `UI_PROMPT` 仅保留为「整体规格说明」注入 system，**不再要求一轮产出全部**。

详细 prompt 与 7 大模块、8 套页面模板、设计令牌请见 notion-91。

### 9.5 设计一键生成串行端点（J8）

新增 `POST /api/projects/:id/design/generate-all`，maxDuration=1800s。按 `["summary", "detail", "api", "db", "ui"]` **严格串行**执行，前一步失败立即 `throw e` 阻断后续：

```tsx
// src/app/api/projects/[id]/design/generate-all/route.ts
const DESIGN_ORDER = ["summary", "detail", "api", "db", "ui"] as const

for (let i = 0; i < DESIGN_ORDER.length; i++) {
  const subtype = DESIGN_ORDER[i]
  results[subtype] = "running"
  ctx.send("progress", { phase: "design-step-start", subtype, results, step: i, total: 5 })
  try {
    const content = await generate(ctx, subtype)
    results[subtype] = "done"
    ctx.send("progress", { phase: "design-step-done", subtype, results, ... })
  } catch (e) {
    results[subtype] = "failed"
    ctx.send("progress", { phase: "design-step-failed", subtype, results, ... })
    throw e  // 不跳后续
  }
}
ctx.send("result", { results, totalMs })
```

**SSE 进度事件类型**：

| 事件 phase | 触发时机 | 携带字段 |
| --- | --- | --- |
| `design-plan` | 任务开始 | `subtypes: string[]`, `results: Record<string, StepResult>` |
| `design-step-start` | 每个 subtype 开始 | `subtype`, `step`, `total`, `results` |
| `design-step-done` | 每个 subtype 完成 | `subtype`, `elapsedMs`, `message` |
| `design-step-failed` | 每个 subtype 失败 | `subtype`, `elapsedMs`, `message`（含错误） |
| `design-ui-stream` | UI 子步骤实时流（shell / 6 page / patch） | `subPhase`, `delta`, `cumulativeBytes` |
| `result` | 全部完成 | `results`, `totalMs` |

### 9.6 Mermaid 流程图主题注入（J9）

`src/lib/markdown/mermaid-theme.ts` 导出 light/dark 双套主题变量，统一所有 mermaid 图表配色：

```tsx
export function injectMermaidTheme(md: string, theme: "light" | "dark"): string {
  const header = theme === "dark" ? MERMAID_DARK : MERMAID_LIGHT
  return md.replace(/```mermaid\s*\n(?:%%\{\s*init:[^}]*\}\s*%%\s*\n)?/g, "```mermaid\n" + header)
}
```

**集成点**：
- `src/components/workbench/MarkdownView.tsx`：渲染前 `useMemo(() => injectMermaidTheme(source, "light"))`
- `src/components/markdown.tsx`：同上
- `src/lib/export/pandoc.ts`：pandoc 转换前注入 light 主题到临时文件
- `scripts/inject-mermaid-theme.ts`：批量回填已有文档，支持 `--dry-run`
- 设计 DB prompt 末尾追加「禁止在 ```mermaid 块内添加 `%%{init:…}%%` 头，主题由渲染端统一注入」
